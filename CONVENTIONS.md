|---|---|---|

| Road to Folding a T-shirt | 1桁から | `#1` |

| Physical AI 研究ノート | 1桁から | `#1` |

- **番号は時間ではなく回数を数える。** 3日空いても次は `#4`。日付連番にしない
- 桁は勝手に増える。最初から揃えない

## YouTubeタイトル

**中身が先、シリーズ名と番号は後ろ。** 区切りは全角縦棒 `｜`。

```
[中身]｜[シリーズ名] #n
```

例:

```
✓ 目の前で動くR1に触ったら、想像と全然違った｜Physical AI 研究ノート #1
✓ 3日溶かした原因がUSBの挿し方だった件｜Failure Log #3
✓ SO-101が初めて物を掴んだ｜Road to Folding a T-shirt #12

✗ Physical AI 研究ノート #1 — 目の前で動くR1に触った日
✗ 【Failure Log #3】3日かけてUSBの挿し方が悪かった
```

理由: タイトル左側が最も読まれる。毎回同じ文字列を置くと読み飛ばされる。

## GitHub

ファイル名は逆。ソート優先で番号が先。

```
build-log/
  build-log-001.md
  build-log-002.md
```

ファイル内の見出しは通常表記。

```markdown
# Build Log #001 — 目の前で動くヒューマノイドに触った日
```

## Failure Log の使い方

**常用しない。** 失敗した日に「それでも出す」ための逃げ道として持っておく。

- 全部に付けると、成功した日に使えなくなる
- ラベルとして常用すると「失敗を売りにしている」感が出て、失敗が演出になる
- 普段は `Road to Folding a T-shirt #n`。何も動かなかった週だけ `Failure Log #n`

## Physical AI 研究ノート の掟

ニュース解説にしない。毎回、最後に必ずこれをやる。

> **で、これは自分のSO-101にとってどういう意味があるのか？**

- 外の情報は、必ず自分の作業に着地させる
- 着地しないニュースは、扱わない
- 着地させる制約があるので、毎日ニュースを追う必要はない

これができていれば、ニュースを扱っても研究室チャンネルのままでいられる。
