# Build Log #002 — 2026-07-20

## 今日やったこと

Claude Codeの基礎機能を、3セッションに分けて体系的に学習した。

### 1. Claude Codeの位置づけ
AI支援開発には大きく3つの形態がある。
- **エディタ支援ツール**：タブ補完のように、コードエディタの中で部分的に支援する
- **アプリケーション自動生成ツール**：プロンプトからアプリの立ち上げまで一括生成する
- **自律型コーディングエージェント**：Issueを立てる形でタスクを渡し、AIがプルリクエストという成果物を返す

Claude Codeが従来のAI支援開発と違う点は、**ターミナル・ファイルシステム・Git・実行環境に直接アクセスできること**。チャット型やタブ補完は「コードの文字列」しか扱えないが、Claude Codeは「コマンドを実行して結果を見て、次の判断をする」というループを自分で回せる。これが自律性の正体。

ハルシネーション（AIが事実でないことを事実として出力する現象）は依然として起きるが、Claude Codeは実行結果を自分で確認できるため、間違いに自分で気づいて修正するループが働きやすい。とはいえ完全ではないので、大きな変更ほど小さいステップに分けて都度確認する運用が有効。

### 2. プラン・モデル・コスト管理

**`/upgrade`**：現在のサブスクリプションプランを別のプランに切り替えるコマンド。Enterpriseプランでは表示されないなど、プランによって出る/出ないが変わる。

**セッションと制限**：メッセージの利用上限は5時間ごとにリセットされるローリングセッション制。最初のメッセージから5時間が1セッションとしてカウントされ、時計のリセット（0時など）とは無関係に進む。これとは別に週単位（7日）の利用制限があり、Claude.aiのチャットとClaude Codeは同じ利用枠を共有している。つまりチャットで使いすぎるとClaude Codeの残量にも影響する。

**`/model [モデル名]`**：使用モデルを切り替え、次回セッション以降のデフォルトとしても保存される。引数なしで実行するとピッカーが開く。
- **Sonnet**：日常的なコーディング作業向け。効率重視
- **Opus**：複雑・難解なタスク、バグ修正向け。Sonnetのおよそ2倍の使用量を消費する代わりに、より高度な複雑タスクに強い
- **Fable**：最も難度が高く、最も長時間かかるタスク向けの最上位モデル。日常のコーディングで使う必然性は薄く、大規模リファクタリングや長時間の自律実行タスクなど「賭け」に近い場面での起用が想定される
- **Haiku**：最速で軽量なタスク向け。単純な調査や即答が欲しい場面に向く

モデル指定は`/model`コマンドの他、環境変数`ANTHROPIC_MODEL`やCLAUDE.mdへの記述でも可能。用途に応じて自動化・固定化できる。

**1Mトークンコンテキストモデル**：モデル名に`[1m]`という接尾辞を付けて指定する、拡張コンテキスト版。従量課金（API課金）でのみ利用可能で、サブスクリプションには含まれない。巨大なコードベース全体の解析や、大量のログファイルをまとめて読ませたい場面で使う。narrow_v1のような大量の試行ログを一括分析させたい場合の選択肢になりうる。

**`/cost`・`/usage`**：どちらも現在の使用量を確認するコマンドで、実質同じ情報を表示する（`/cost`は`/usage`のエイリアスという扱い）。API課金ユーザーはコスト（金額）を、サブスクリプションユーザーは利用枠の消費量を確認する用途になる。個人開発では、まずサブスクリプションから始めて必要に応じてAPI課金に移行するのが一般的な流れ。

### 3. セットアップ・環境

**インストール**：Macの場合`brew`でインストール可能。Node.js経由でのnpmインストールにも対応（公式ドキュメント：https://code.claude.com/docs/ja/setup#install-with-npm）。

**gh（GitHub CLI）**：GitHubが公式に提供するコマンドラインツール。PRの作成・レビュー・issue操作などをターミナルから直接行える。Claude Codeとの連携はオプションだが、入れておくとPR関連の自動化（CIが落ちた時の自動修正など）が可能になる。インストールはOSごとに`brew install gh`（Mac）、`winget install --id Github.cli`（Windows）、`sudo apt install gh`（Linux）。導入後`gh auth login`で認証する。

**IDE連携**：VS Code、JetBrains系IDEと連携可能。エディタ上でClaude Codeの提案や差分を直接確認しながら作業できる。

**`claude update`**：Claude Code自体をアップデートするコマンド。プランのアップグレード（`/upgrade`）とは別物。

**`claude terminal-setup`（Mac）**：Terminal.appの設定を自動調整し、Claude Codeとの相互運用性を改善するコマンド。実行後、Option+Enterで改行入力ができるようになり、ターミナル内での快適性が向上する。内部的には「Use Option as Meta key」を有効化し、聴覚ベルを無効化する。実行後はTerminal.appの再起動が必要。

### 4. CLAUDE.mdとメモリ

**`/init`**：プロジェクトのルートに`CLAUDE.md`を生成する。プロジェクトの文脈・規約をClaude Codeに常時読み込ませるための基盤ファイル。

**`@`記法**：メッセージ内で`@ファイル名`と書くことで、特定のファイルを指定して読み込ませたり参照させたりできる。CLAUDE.md内でも`@`記法で他のファイルを参照可能。ただし、コードスパン（バッククォート` `` `で囲んだ範囲）内に書くと評価されず、ただの文字列として扱われる点に注意。

**CLAUDE.mdの配置**：プロジェクトルートだけでなく、サブディレクトリにも置ける。ルート直下のCLAUDE.mdはセッション開始時に常に読み込まれるが、サブディレクトリのCLAUDE.mdはClaudeがそのディレクトリ配下のファイルを扱う際にオンデマンドで読み込まれる仕組み。モノレポや複数プロジェクトが混在する構成で有効。

**メモリの二重構造**：
- `#`から始めるメッセージを入力すると、その内容をプロジェクト用またはユーザー用のCLAUDE.mdに追記するかを選べる「クイック追加」の仕組みがある
- これとは別に「Auto Memory」という自動学習の仕組みが存在する。CLAUDE.mdが「人間が明示的に書く指示」であるのに対し、Auto Memoryは「Claudeが会話の中で自分で気づいたこと（うまくいったビルドコマンド、繰り返し起きたミス、プロジェクト固有のパターンなど）」を、CLAUDE.mdとは別のファイル（プロジェクトごとのメモリディレクトリ内）に自動で書き溜めていく。デフォルトでON。`/memory`コマンドで現在読み込まれているCLAUDE.mdやAuto Memoryの状態を確認・編集できる

### 5. セッション操作

- **ESC**：現在実行中の指示をキャンセル
- **ESC2回**：会話とコードの状態を過去のチェックポイントまで巻き戻す（`/rewind`と同等の機能）
- **`claude --continue` / `claude -c`**：直前のセッションからそのまま再開
- **`claude --resume`（起動時）／`/resume`（対話モード内）**：過去のセッション一覧から選んで再開
- **Ctrl+R**：会話履歴を検索
- **`/exit`**：Claude Codeとの対話を終了

### 6. 許可モード

Shift+Tabを押すと、コマンド実行やファイル編集をどこまで自動承認するかのモードが切り替わる。基本サイクルは以下の3つ：
- **default（manual）**：ファイル読み込みは自由に行うが、編集やコマンド実行の前に毎回確認を求める。最も慎重なモード
- **acceptEdits**：ファイル編集は自動承認されるが、作業ディレクトリ外への影響を持つ操作などは引き続き確認が入る
- **plan**：Claudeが調査・計画のみを行い、実際の変更は加えないモード。複雑な変更に着手する前の設計確認に向く

これに加えて、条件付きで以下の2モードがサイクルに加わる：
- **auto**：アカウントが条件を満たしている場合に出現。分類器が各操作を判定し、読み取り専用の操作などは確認なしで自動承認する
- **bypassPermissions**：`--dangerously-skip-permissions`フラグ付きで起動した場合のみ利用可能。全ての操作が無確認で実行される「安全ネットなし」モード。`.git`や`.claude`など一部の保護対象パスへの書き込みだけは、このモードでも確認が必要な場合がある

初回のコマンド実行時には確認プロンプトが出るが、「今後も許可」を選ぶと、その許可ルールが`.claude/settings.local.json`に保存されていき、次回以降は確認なしで実行されるようになる。

### 7. 拡張思考モード

Option+T（Mac）でトグルできる、Claudeが応答前に内部で深く考える「拡張思考」のON/OFF切り替え。ONにすると複雑な設計判断・微妙なバグ・多段階の計画で回答の質が上がるが、応答は遅くなり、思考トークン分のコスト（利用枠の消費）も増える。単純な修正作業ではOFFのままの方が効率的。

### 8. MCPサーバー

**`claude mcp list`**：現在接続されているMCPサーバーの一覧を確認する。

**`claude mcp add <サーバー名> -- <起動コマンド>`**：新しいMCPサーバーを追加する。例：`claude mcp add context7 -- npx -y @upstash/context7-mcp`。追加後は、プロンプト内で「〇〇を作ってください use context7」のように明示的に使用を指示する運用になる。

**Context7**：ライブラリやフレームワークの最新ドキュメントをリアルタイムで参照できるMCP。学習データのカットオフ後に登場した仕様やAPIの変更に対応しやすくなる。

**Playwright**：Microsoftが2020年に公開したオープンソースのブラウザテスト自動化ツール（Seleniumに近い立ち位置）。このMCPを導入すると、E2Eテスト（エンドツーエンドテスト）の自動化をClaude Codeに任せられる。完全無料・オープンソース。

**Serena**：コードベースに対するClaudeの言語理解精度を高め、より正確なコード操作を可能にするMCP。

## 気づき・考察

### (1) 「教材の鮮度」問題は、これからも繰り返し起きる
今日わかった一番大きいことは、**ネット上の解説記事は書かれた瞬間から陳腐化していく**という前提を持つ必要があるということ。Claude Codeは更新サイクルが速く、コマンド名やモデル名は数ヶ月で変わりうる。これは一度直して終わりの話ではなく、Project Physical AI全体の発信ルール（事実・予測・ビジョンの峻別）と同じ構造の問題として運用に組み込むべき。動画や記事でClaude Codeの機能に言及する際は、収録・執筆時点のバージョンを明記する運用にしてもいいかもしれない。

### (2) 許可モードの理解は、今後SO-101制御コードを書かせる上で実務的に重要になる
`default → acceptEdits → plan`のサイクルと、`bypassPermissions`が意図的にサイクル外に置かれている設計は、**物理的なロボットを動かすコードを書かせる際の安全設計としてそのまま参考になる**。ソフトウェアの誤操作はファイルが壊れるだけだが、SO-101の制御コードの誤操作は実機が予期しない動きをするリスクがある。今後Claude CodeにSO-101の制御スクリプトを書かせる場面では、acceptEditsで確認しながら進め、bypassPermissionsは使わないという運用方針を意識的に決めておいた方がよさそう。

### (3) メモリ機構の二重構造は、narrow_v1のデバッグに直結しうる
CLAUDE.md（人間が明示的に書く指示）とAuto Memory（Claudeが自動で学習していく気づき）の役割分担がわかった。narrow_v1のデータ収集で「箱位置固定・ブロック位置ランダム」という条件を試行錯誤している最中は、Claudeが気づいた失敗パターン（例：特定の角度で把持失敗しやすい、特定のライティングで認識精度が落ちる）がAuto Memoryに自然に蓄積されていく可能性がある。次回のセッションで`/memory`を開いて、実際にどんな内容が自動保存されているかを確認する価値がありそう。

### (4) MCPサーバー（特にPlaywright・Serena）はProject Physical AIの周辺タスクに応用できる
- Playwrightは直接ロボット制御には関係ないが、将来Build Logや進捗をWeb上で可視化するダッシュボードのようなものを作る場合、テスト自動化として使える
- Serenaはコード精度を上げる仕組みなので、LeRobot・ROS2まわりのPythonコードを書かせる際に有効化して比較してみるのは面白い検証企画になりそう（企画ストック3番「制約型」的な立ち位置）

### (5) コスト管理の理解は、ハードウェア投資の意思決定にも間接的に関わる
サブスクリプションとAPI課金の違い、5時間セッション制、週次リセットの仕組みを理解したことで、今後Claude Codeにどれくらいの作業量を任せられるかの見積もりが立てやすくなった。BTO PC（RTX 5060 Ti）着荷後、模倣学習のデータ収集・前処理・モデル訓練スクリプトをClaude Codeにまとめて書かせる場面が増えるはずなので、セッション制限に引っかかって作業が中断するリスクを事前に把握できたのは実務的に有益。
